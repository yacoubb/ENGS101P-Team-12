Programming Principles, Logic and Verification

Introduction

Programming Principles, Logic and Verification is an area of research dedicated to the development of mathematical concepts and theorems which are used to help in other areas of computer science, such as automated debugging and also the underlying logic of early computer system development.

History

  • 340BC

    Aristotle started work on a philosophic field, term logic or Aristotelian logic, which formed the roots for formal logical reasoning.[1]

  • 1800s

    Concurrent verification developed out of earlier work on railroads and telegraphy. This work arose to address the question of how to handle multiple trains on the same railroad system and how to handle multiple transmissions over a given set of wires. (Time division multiplexing)[2]

  • 1850

    George Boole's Boolean Algebra was a complete system on which computational processes could be mathematically modelled.[3]

  • 1879

    Frege's Begriffsschrift introduced the idea of modern predicate logic. Predicate logic uses quantified variables in place of non-logical objects so that more general cases can be created. (i.e. propositions such as Socrates is a man are written in the form "there exists X such that X is Socrates and X is a man and there exists is a quantifier while X is a variable.") This subtle difference was integral in the development of more complex computational logic.[4]

  • 1936

    Alan Turing proves that a general algorithm to solve the halting problem for all possible program-input pairs cannot exist. A key part of the proof was a mathematical definition of a computer and program, which became known as a Turing machine; the halting problem is undecidable over Turing machines. This was a big step forward for the field of logic and verification.[5]

  • 1940

    The enigma code cracking machine called a bombe, code-named Victory, was installed in March 1940. This machine relied heavily on the mathematical theorems and models created the equivalent of the PPLV group at the time.[6]

  • 1965

    The academic study of concurrent algorithms started in the 1960s, with Dijkstra credited with being the first paper in this field, identifying and solving mutual exclusion.[7]

Partial order semantics for concurrent program verification

Aim of research

The main purpose of this research area is to solve the concurrent programming errors by achieving the automatic verification with high scalability and then find the automated techniques that can debug the code without executing it.

Problem to be Solved

Multiprocessor systems are computers with more than one central processor. Today, more than ever, there is a growing trend of using multiprocessor systems. Although used in most modern devices, they are a cause of concurrent programming errors.

An Example of the Project

PostgreSQL, one of the most popular databases, is relied on by many websites. Unfortunately, a concurrency bug was recently found in it. The bug was particularly hard to observe because it only occurred on a not-so-widely-used multicore machine —IBM Power. [7]

Ideal Solution

The best way to solve issues like this is to find automated techniques that can debug the code without having to execute it.

Impact of Research

It's imperitive that in systems used in stuff like airplanes and life support machines, crashes don't occur. Since these systems are so complex nowadays and require multi-core processors, they use concurrent software to operate. It follows that the algorithms used are completely foolproof at a theoretical level.

Identifiers or the Field's Importance

  • IBM Austin: Derek Williams, a lead engineer in collaboration with the IBM research group in Texas indicates that the project will have a very large potential impact if successful.
  • Microsoft : MSR — Microsoft Research labs in Cambridge, Redmond heavily influence the project.
  • Monoidics, a London based SME is very interested in general verification and effective solution to the concurrency programming errors.

Program Verification Techniques for Understanding Security Properties of Software

"Program verification is when formal, mathematical techniques are used to debug software and software specifications." - Robert S. Boyer

Aim of research

To develop automatic program verification methods that help security engineers to understand software that they have not written themselves.

1. Design and construction of COWL (Confinement with Origin Web Labels)[1]

The vast majority of JavaScript code libraries being available online nowadays make it sometimes difficult to distinguish code created for malicious purposes. Many web developers may thus use such code in their applications, leading to the distribution of it further, without having any malicious purposes on the first place though. Of course, in thus way the privacy of any user of the application is at risk, as such malicious code may be used to collect personal data, which are sensitive by nature.
The relevant UCL research team in cooperation with Mozilla, Google, Stanford, and UCSD have developed “Confinement with Origin Web Labels” (COWL), which is a technique used in many internet browser that blocks malicious applications from being able to collect and transmit sensitive data to a third party.

Impact

  • COWL is about to become a standard with the World Wide Web Consortium (W3C). W3C is in charge of standardising how the internet operates to a great extent and therefore many of the features it standardises are included in servers, web browsers, websites.
  • Once the standardising of COWL is completed, it is going be embedded by default in Google Chrome and Mozilla Firefox, two of the most widely used internet browsers.

2. Design and construction of the G8 system

Developers are humans and it is almost unavoidable to make errors. Such errors may create minor but serious vulnerabilities in server-side applications that may lead to the disclosure of sensitive data, due to wrong access control checks for instance. Server-side applications operate in a server computer instead of operating locally on the clients’ one.

That is why G8 has been developed. G8 is a policy system that operates during the run-time of the Node.js JavaScript framework for server-side programs and ensures that:

  • No unsafe operations take place
  • Sensitive information is not disclosed to unauthorised people because of program bugs
Impact

A complete version of G8 has been created and it has been proved to successfully prevent exploits of vulnerabilities in some widely-used server-run programs using Node.js.
